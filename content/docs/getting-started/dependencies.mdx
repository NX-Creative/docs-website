NX Creative scripts are built on top of the
[Overextended](https://overextended.dev) ecosystem. This is an opinionated
choice: we don't ship fallback paths for `qb-inventory`, `qb-target`, or
`mysql-async`. If a script uses an inventory, it expects `ox_inventory`.

## Why ox?

Three reasons, in order:

1. **Actively maintained.** The ox resources see regular releases and
   respond to FiveM runtime changes quickly.
2. **Consistent API shape.** `ox_lib` gives us notifications, contexts,
   menus, and callbacks with a single import. Same with target zones.
3. **Item & data integrity.** `ox_inventory` treats items as first-class
   and stores state in MySQL. Building on top of it means an NX script
   can trust what's in a player's inventory without defensive re-queries.
